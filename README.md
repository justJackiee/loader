# 🔴 Shellcode Loader

A Windows shellcode loader for learning malware development fundamentals. This project demonstrates encrypted shellcode injection using **direct syscalls** (Hell's Gate / Halo's Gate), sandbox evasion, AES-256 encryption, remote process injection, and **DLL side-loading**.

## 📁 Project Structure

```
Loader/
├── loader.c            # EXE loader (syscall-based injection into notepad.exe)
├── proxy.c             # Proxy DLL loader (version.dll side-loading)
├── syscalls.h          # Hell's Gate structs, hash constants, Nt* prototypes
├── syscalls.c          # SSN resolver (Hell's Gate / Halo's Gate) + Nt* wrappers
├── syscalls_asm.asm    # x64 assembly syscall stub (NASM)
├── exports.def         # Export definition file (clean names for proxy DLL)
├── aes.c / aes.h       # TinyAES library (AES-256-CBC)
├── resource.h / .rc    # Resource ID definitions + script
├── builder.py          # Automated build script (encrypts + assembles + compiles)
├── payload.bin         # Your shellcode (msfvenom / Sliver)
└── README.md
```

> **Note:** `key.h`, `payload_enc.bin`, `resource.o`, and `syscalls_asm.o` are auto-generated by `builder.py` during the build and cleaned up afterward.

## 🔧 Build & Compile

### Prerequisites

| Tool | Purpose |
|------|---------|
| `x86_64-w64-mingw32-gcc` | Cross-compile for Windows |
| `x86_64-w64-mingw32-windres` | Compile resource scripts |
| `nasm` | Assemble x64 syscall stub |
| `python3` + `pycryptodome` | Run builder / AES encryption |

### Automated Build (Recommended)

```bash
# Build EXE loader (direct syscall injection)
python3 builder.py payload.bin

# Build proxy DLL (DLL side-loading)
python3 builder.py payload.bin --proxy

# Custom output name
python3 builder.py payload.bin -o agent.exe

# Keep intermediate files
python3 builder.py payload.bin --keep
```

**Build pipeline:**
1. Generate random AES-256 key (32 bytes) + IV (16 bytes)
2. Encrypt `payload.bin` with AES-256-CBC + PKCS7 padding → `payload_enc.bin`
3. Write `key.h` with key/IV as C byte arrays
4. Compile resource script → `resource.o`
5. Assemble syscall stub → `syscalls_asm.o` (NASM)
6. Compile everything:
   - **EXE:** `loader.c` + `syscalls.c` + `aes.c` + `syscalls_asm.o` + `resource.o` → `loader.exe`
   - **DLL:** `proxy.c` + `syscalls.c` + `aes.c` + `syscalls_asm.o` + `resource.o` + `exports.def` → `version.dll`
7. Clean up intermediates

### Manual Compilation

```bash
# Assemble syscall stub
nasm -f win64 syscalls_asm.asm -o syscalls_asm.o

# Compile resource
x86_64-w64-mingw32-windres resource.rc -o resource.o

# EXE loader
x86_64-w64-mingw32-gcc loader.c syscalls.c aes.c syscalls_asm.o resource.o -o loader.exe -s

# Proxy DLL
x86_64-w64-mingw32-gcc proxy.c syscalls.c aes.c syscalls_asm.o resource.o exports.def -shared -o version.dll -s
```

## 📖 Theory

### Week 11: Direct Syscalls — Hell's Gate / Halo's Gate

#### The Problem: Userland Hooking

EDRs (CrowdStrike, SentinelOne, etc.) inject a DLL into every process and overwrite the first bytes of `ntdll.dll` functions (like `NtAllocateVirtualMemory`) with a `JMP` to their analysis engine. When your loader calls `VirtualAlloc`, the EDR intercepts the call, inspects arguments, and decides whether to allow or kill it.

```
Your Loader → VirtualAlloc (kernel32.dll)
            → NtAllocateVirtualMemory (ntdll.dll)  ← EDR JMP hook here!
            → EDR Analysis Engine (inspect & decide)
            → syscall → Kernel
```

#### The Solution: Direct Syscalls

By manually preparing the registers and executing the `syscall` instruction ourselves, we **skip the hooked code entirely**:

```
Your Loader → HellDescent() (our asm stub)
            → mov r10, rcx
            → mov eax, SSN       ← Syscall Service Number
            → syscall             ← Jump straight to Ring 0
            → Kernel              ← EDR hooks are never touched
```

#### Hell's Gate: SSN Resolution

The Syscall Service Number (SSN) is the ID the kernel uses to identify which function to run. It changes between Windows builds, so we resolve it **dynamically** at runtime.

If `ntdll.dll` is **not hooked**, the function starts with a known pattern:
```asm
mov r10, rcx    ; 4C 8B D1
mov eax, <SSN>  ; B8 xx xx 00 00    ← SSN is right here
```

We read the bytes at the function address and extract the SSN.

#### Halo's Gate: Handling Hooked Functions

If the EDR has hooked the function (first byte is `0xE9` = JMP), we scan **neighboring syscall stubs** (±32 bytes). Since SSNs are sequential, if a neighbor's SSN is `N`, ours is `N ± offset`.

#### API Replacement Summary

| Before (Hooked by EDR) | After (Direct Syscall) |
|---|---|
| `OpenProcess` | `NtOpenProcess` via `SysNtOpenProcess()` |
| `VirtualAllocEx` | `NtAllocateVirtualMemory` via `SysNtAllocateVirtualMemory()` |
| `WriteProcessMemory` | `NtWriteVirtualMemory` via `SysNtWriteVirtualMemory()` |
| `VirtualProtectEx` | `NtProtectVirtualMemory` via `SysNtProtectVirtualMemory()` |
| `CreateRemoteThread` | `NtCreateThreadEx` via `SysNtCreateThreadEx()` |

#### Injection Flow (loader.c)

```
┌──────────────────────────────────────────────────────┐
│ 0. InitializeVxTable()                               │  Hell's Gate: resolve all SSNs
│ 1. SysNtOpenProcess()                                │  Open handle to notepad.exe
│ 2. FindResource → AES Decrypt                        │  Load & decrypt payload locally
│ 3. SysNtAllocateVirtualMemory()                      │  Allocate RW memory in notepad
│ 4. SysNtWriteVirtualMemory()                         │  Write decrypted shellcode
│ 5. SysNtProtectVirtualMemory()                       │  Flip RW → RX
│ 6. SysNtCreateThreadEx()                             │  Execute shellcode in notepad
└──────────────────────────────────────────────────────┘
```

### Week 9: Proxy DLL / DLL Side-Loading

DLL side-loading abuses the Windows DLL search order. When an app calls `LoadLibrary("version.dll")`, Windows searches the **app's directory first** before `System32`. By placing a malicious `version.dll` next to the app, Windows loads ours instead.

**Why version.dll?** Many apps load it, it only has 17 exports, and it's not a "Known DLL" — so the app directory is searched first.

The proxy DLL **forwards all 17 exports** to the real `C:\Windows\System32\version.dll` via runtime resolution (`LoadLibrary` + `GetProcAddress`), then spawns a background thread for the payload.

### Sandbox Evasion

| Check | What it Does | Why |
|-------|-------------|-----|
| **RAM check** | Exits if total RAM < 4 GB | Sandboxes often have low memory |
| **Cursor movement** | Checks if cursor moved after 1s | Automated analysis has no real user |

### AES-256-CBC Encryption

- **Builder** (Python): encrypts payload with `pycryptodome` + PKCS7 padding
- **Loader** (C): decrypts in-memory using TinyAES after loading from .rsrc
- Payload is **never stored decrypted on disk**

### Why RW → RX?

Allocating `RWX` memory is a red flag. Better approach:
1. Allocate as **RW** (normal)
2. Write + decrypt shellcode
3. Flip to **RX** (execute-only, no write)
